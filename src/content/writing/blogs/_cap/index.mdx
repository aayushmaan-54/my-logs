## Introduction
You might have hered that you need to choose two out of three properties in a distributed system, but what are those properties? And why do we need to choose only two? but the framing that even the theorem's creator, Eric Brewer, described as misleading in a follow-up paper 12 years later (2012).

## What is the CAP Theorem?
- Consistency: Every read receives the most recent write or an error.
- Availability: Every request receives a response, without guarantee that it contains the most recent write.
- Partition Tolerance: The system continues to operate despite an arbitrary number of messages being dropped (or delayed) by the network between nodes.
- In distributed systems, we always need P(partition tolerance) because we cannot guarantee that the network will always be reliable as things will fail.

## Choosing between C and A
Peple generally say that P is necessary and we have to choose between C and A, but in reality its not a binary choice (means we pick one and ignore other) but its spectrum of value.
Moslty avaalibilty is measure in 9's (eg 99.99%, 99.95% uptime)
Consistency is also spectrum, we can have strong consistency, eventual consistency, causal consistency, etc. and we can choose the level of consistency we want based on our use case.

## Real world systems and CAP
Brewer's updated framing: 
Partition tollerence in relaity is rare but constant, and most of time system is in normal state, so we can have both consistency and availability, but when partition happens we have to choose which one we want to sacrifice, but we can also have a system that can degrade gracefully and provide some level of consistency and availability even during partition.
Real system dont live in binary world of picking C or A in triangle but they live inside the triangle and choose diffrent levels of C, A and P based on their use case and requirements.

## Designing for Partition
- Most engineers design things for 2 state everyhting working fine and everyhitng is broken that is binary thinkng Eric Brewer says distributed systems have 3rd state:
NORMAL  →  PARTITION  →  RECOVERY

1. Normal state: everything is working fine, no partitions, no failures.
- In this state, all nodes can communicate. Engineering is focused on the "happy path":
  - Goal: Optimize for speed, low latency, and high consistency.
  - Tools: Load balancers, read replicas, and caching are all tuned for maximum performance because the network is reliable.

2. Partition Mode (The "Strategic" Failure): This is where most engineers mess up. Network issue → timeout → throw error → retry → chaos. System treats it like: “Something broke randomly”.
Instead What Brewer suggests Treat partition as a first-class state means you detect it intentionally and you switch system behavior intentionally.
Instead if you defining one global rule for your system you define what each feature should do during partition/

## CRDTs (Conflict-free Replicated Data Types)
- In CAP we choose b/w C & A but in reality we can have both C & A by using CRDTs.
- lets say user a update a value and at same time user b update same value so for that old solution will be locks or pick winner but with CRDTs we can merge both updates without losing any data and without needing locks.
- CRDT ask can we design data structure in such a way that we can merge different versions of it without losing any data and without needing locks.
- it dont force but design for safe merging of data.
- like in above example node a update value and node b update value and later we can merge both updates without losing any data and without needing locks.
- after sync: all node will have same data and it will be consistent and available even during partition.
- for data structure to be DS it must follow these rules:
  - Commutative: order of operations should not matter (1 + 2 = 2 + 1).
  - Idempotent: applying the same operation multiple times should have the same effect as applying it once.
- This is used in google docs where multiple users can edit the same document at the same time and later their changes are merged without losing any data and without needing locks.
- Youtube views all server have diffrent view count but later they merge all views and update the view count without losing any data and without needing locks.
- CRDT shine on case when we have simple counters like views or likes, colloborative docs, sets/list (eg: shopping cart) but for complex data structure like bank account balance it is not suitable because we need strong consistency and we cannot merge different versions of balance without losing data or needing locks.

## The Saga Pattern 
- Lets take atm example in this we accept -> bound -> log -> compensate.
- formally described in 1987 by Hector Garcia-Molina and Kenneth Salem in their paper "Sagas".
- In traditonal system we have locks commit or rollback system used in DB but in distributed system we cannot use locks because it can lead to deadlocks and it can also lead to unavailability during partition.
- So instead of locks we can use saga pattern where we break down a transaction into a series of smaller, independent steps (sagas) that can be executed & committed in a distributed manner.
- if one step fail in middle of sagas transaction we can compensate for that step by executing a compensating action that undoes the effects of the failed step, allowing the system to maintain consistency without needing locks.
- for example we have these steps commited charge card -> reserve item -> but shipping failed due to invalid adress so we can compensate for that by refunding the card and releasing the reserved item.

## Local first architecture
- In traditional client-server architecture, the client relies on the server for data storage and processing, which can lead to issues with availability and consistency during network partitions.
- In a local-first architecture,the client maintains a local copy of the data and can operate independently of the server, allowing for better availability and consistency even during network partitions.
- The client can synchronize changes with the server when the network is available, and can also synchronize with other clients directly using peer-to-peer communication, allowing for a more resilient and responsive user experience.
- This approach is particularly beneficial for applications that require offline functionality or need to operate in environments with unreliable network connectivity, such as mobile apps or collaborative tools.
- It flip on partition head because in traditional client-server architecture during partition we have to choose between C and A but in local-first architecture we can have both C and A even during partition because the client can operate independently of the server and can synchronize changes with the server when the network is available.
