## Introduction

Ever felt limited while coding on Windows?
Like things work, but not in the smooth way you see Linux users working.  
At the same time, switching to Linux full-time is hard because some software only runs on Windows, or you're into gaming like me, and Windows still has the best support for that (typical Microsoft stuff).  
That's where **WSL2** comes in.  
In this blog, I'll show you how to set up **WSL2** to run Linux inside Windows, configure **ZSH** to make your terminal cleaner and more powerful, and set up multiple **GitHub SSH keys** so you can easily manage different accounts.  
This setup gives you the best of both worlds.  
Linux for development and Windows for everything else (especially gaming :<).

## Installing WSL2

Before installing **WSL2** on your Windows machine, there are a few things you need to have:

- Windows 10 (Build 19041+) or Windows 11 (Obviously)
- Administrator access
- Virtualization enabled in BIOS/UEFI (Follow any tutorial its pretty easy)
- Windows Terminal
- VS Code (env-var should exist & install WSL extension by Microsoft)

### Installing WSL2 & Ubuntu

Open **PowerShell as Administrator** and run:

```bash
wsl --install
```

This command will:

- Enable WSL feature
- Enable Virtual Machine Platform
- Install Ubuntu (default distro)
- Set WSL2 as default version

Restart your computer when prompted.  
After restart the terminal will automatically open **Ubuntu** in **WSL2** and you will be prompted to create a *username* and *password* — remember them well as they will be needed in the future.  

Few commands to remember for WSL2 that might be helpful:

```bash
# Start WSL
wsl

# See available distributions
wsl --list --online

# Start specific distro
wsl -d Ubuntu-24.04

# Check WSL version
wsl --version

# Install Ubuntu 24.04 LTS
wsl --install -d Ubuntu-24.04

# Uninstall Ubuntu 24.04 LTS
wsl --unregister Ubuntu-24.04

# Stop all WSL instances (Very useful when WSL feels buggy)
wsl --shutdown

# Set WSL default distro
wsl --set-default Ubuntu

# Update WSL
wsl --update
```

If you face problems installing WSL you can also install it via **Microsoft Store**, but I would recommend installing it via *CLI* as it gives more control and you know better what is going on.

#### Updating Linux

After setting up username and password you will be inside Ubuntu. Next you need to update the source list and upgrade the distro:

```bash
sudo apt update && sudo apt upgrade -y
```

Enter your password when prompted.

You can create a `.wslconfig` file where you can allocate WSL resources such as memory, storage or processor.

#### Setting Up Zsh + Starship

We'll use **Zsh** as our shell with **Starship** as the prompt — minimal, fast, and looks great with JetBrains Mono.

##### 1. Install JetBrains Mono Font

The font needs to be installed on **Windows** (not inside WSL), so that Windows Terminal can use it.

Download JetBrains Mono from the official site: [jetbrains.com/lp/mono](https://www.jetbrains.com/lp/mono/)

Extract the zip, select all `.ttf` files inside the `fonts/ttf/` folder, right-click and choose **Install for all users**.

After installing, open Windows Terminal settings and set the font:

> Settings → Profiles → Ubuntu → Appearance → Font face → **JetBrains Mono**

##### 2. Install Zsh

```bash
sudo apt update
sudo apt install zsh -y
chsh -s $(which zsh)
```

Close and reopen your terminal (or restart WSL) for the shell change to take effect.

##### 3. Install Plugins Manually

```bash
mkdir -p ~/.zsh
git clone https://github.com/zsh-users/zsh-autosuggestions ~/.zsh/zsh-autosuggestions
git clone https://github.com/zsh-users/zsh-syntax-highlighting ~/.zsh/zsh-syntax-highlighting
```

##### 4. Install Starship

```bash
curl -sS https://starship.rs/install.sh | sh
```

##### 5. Configure .zshrc

Create or open your `.zshrc`:

```bash
nano ~/.zshrc
```

Paste the full config:

```zsh
HISTSIZE=10000
SAVEHIST=10000
HISTFILE=~/.zsh_history
setopt histignorealldups sharehistory
setopt hist_ignore_space        # commands prefixed with a space won't be saved (useful for secrets)
setopt auto_cd                  # type a directory name to cd into it
setopt correct                  # suggest corrections for mistyped commands
setopt interactive_comments     # allow comments in interactive shell (# like this)

bindkey -v
KEYTIMEOUT=1

# Cursor shape for vi mode (block in normal, beam in insert)
function zle-keymap-select {
  if [[ $KEYMAP == vicmd ]]; then
    echo -ne '\e[2 q'
  else
    echo -ne '\e[6 q'
  fi
}
zle -N zle-keymap-select
echo -ne '\e[6 q'

# Restore text search in vi mode (ctrl+r)
bindkey '^R' history-incremental-search-backward

autoload -Uz compinit
compinit

zstyle ':completion:*' auto-description 'specify: %d'
zstyle ':completion:*' completer _expand _complete _correct _approximate
zstyle ':completion:*' menu select=long
zstyle ':completion:*' matcher-list '' 'm:{a-z}={A-Z}' 'm:{a-zA-Z}={A-Za-z}' 'r:|[._-]=* r:|=*'
zstyle ':completion:*' verbose true
zstyle ':completion:*' list-prompt '%SAt %p: Hit TAB for more%s'
zstyle ':completion:*' select-prompt '%SScrolling: %p%s'

eval "$(dircolors -b)"
zstyle ':completion:*:default' list-colors ${(s.:.)LS_COLORS}
zstyle ':completion:*:kill:*' command 'ps -u $USER -o pid,%cpu,tty,cputime,cmd'

# Aliases
alias ls='ls --color=auto'
alias ll='ls -lah'
alias grep='grep --color=auto'
alias gs='git status'
alias gd='git diff'
alias gl='git log --oneline --graph --decorate'
alias ..='cd ..'
alias ...='cd ../..'

# Arrow key history search
autoload -Uz up-line-or-beginning-search down-line-or-beginning-search
zle -N up-line-or-beginning-search
zle -N down-line-or-beginning-search
bindkey '^[[A' up-line-or-beginning-search
bindkey '^[[B' down-line-or-beginning-search

# Auto-start ssh-agent and add keys on new session
if [ -z "$SSH_AUTH_SOCK" ]; then
  eval "$(ssh-agent -s)" > /dev/null
  ssh-add ~/.ssh/id_personal 2>/dev/null
  ssh-add ~/.ssh/id_office 2>/dev/null
fi

# Git wrapper — auto-routes clone to correct GitHub account based on folder
git() {
  if [[ "$1" == "clone" && "$PWD" == "$HOME/office"* ]]; then
    shift
    local rewritten=("${@/git@github.com:/git@github-office:}")
    command git clone "${rewritten[@]}"
  else
    command git "$@"
  fi
}

source ~/.zsh/zsh-autosuggestions/zsh-autosuggestions.zsh
source ~/.zsh/zsh-syntax-highlighting/zsh-syntax-highlighting.zsh
eval "$(starship init zsh)"
```

Apply it:

```bash
source ~/.zshrc
```

## WSL File Structure

In WSL, Windows drives are mounted at `/mnt/`:

```bash
cd /mnt/c/Users/YourUsername
```

To access your Linux distro files you can either type this in the address bar of Windows Explorer:

```
\\wsl.localhost\
```

Or you can see Linux on the left side sidebar and go from there.

- Set starting directory: `\\wsl.localhost\Ubuntu\home\YourUsername`

## Git Multiple Account Setup

In this section, we'll set up multiple Git identities (office and personal) using SSH in a way that **does not require changing the clone command every time**.  

Normally, when using multiple SSH keys, you have to clone repositories like this:

```bash
git clone git@github-office:username/repository.git
```

This works, but it's annoying to remember and easy to mess up.

In this setup, we'll configure things so that you can simply run the normal clone command:

```bash
git clone git@github.com:username/repository.git
```

…and Git will automatically use the correct SSH key and identity based on which folder you're in.  
No aliases. No custom clone commands. No manual switching.

### Folder Rule (Important)

To make this work smoothly, we follow one simple rule:

- All **office** projects live inside `~/office/`
- Everything **outside** this folder is treated as personal work

Based on this, Git automatically picks the correct identity and SSH key — both at clone time and for all subsequent operations like push, pull, and fetch.

<Callout type="info">
Create your `~/office/` directory before starting the setup and always clone work repos from inside it.
</Callout>

### Why This Is Tricky

Git's `includeIf "gitdir:..."` config rule only activates **after** a `.git` directory exists inside the matched path. During `git clone`, there is no `.git` directory yet — Git hasn't created it — so the conditional block never loads and the wrong SSH key gets used.

We solve this with a small shell function in `.zshrc` that intercepts `git clone` and rewrites the URL before Git ever touches the network.

### Setup

#### 1. Generate SSH Keys

```bash
ssh-keygen -t ed25519 -C "personal@gmail.com" -f ~/.ssh/id_personal
ssh-keygen -t ed25519 -C "office@company.com" -f ~/.ssh/id_office
```

Press Enter to skip the passphrase, or set one for extra security.

#### 2. Set Permissions

```bash
chmod 600 ~/.ssh/id_personal ~/.ssh/id_office
```

#### 3. Add Keys to SSH Agent

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_personal
ssh-add ~/.ssh/id_office
```

#### 4. Configure SSH

```bash
nano ~/.ssh/config
```

Add:

```
# Personal GitHub
Host github-personal
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_personal
    AddKeysToAgent yes

# Work GitHub
Host github-office
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_office
    AddKeysToAgent yes
```

```bash
chmod 600 ~/.ssh/config
```

#### 5. Add Public Keys to GitHub

```bash
cat ~/.ssh/id_personal.pub  # Copy to personal GitHub account
cat ~/.ssh/id_office.pub    # Copy to work GitHub account
```

Add these public keys to each GitHub account:
**Settings → SSH and GPG Keys → New SSH Key**

#### 6. Test Connections

```bash
ssh -T git@github-personal   # should say: Hi <personal-username>!
ssh -T git@github-office     # should say: Hi <work-username>!
```

#### 7. Configure Git Identities

**Global config (personal as default):**

```bash
git config --global user.name "Your Name"
git config --global user.email "personal@gmail.com"
git config --global init.defaultBranch main
```

Then open `~/.gitconfig` and add the `includeIf` rule at the bottom:

```bash
nano ~/.gitconfig
```

```ini
[includeIf "gitdir:~/office/"]
    path = ~/.gitconfig-office
```

<Callout type="warning">
The path in `includeIf` must end with `/` otherwise it won't match.
</Callout>

**Work config:**

```bash
nano ~/.gitconfig-office
```

```ini
[user]
    name = Your Name
    email = office@company.com
```

#### 8. The Clone-Time Fix (The Key Part)

The `includeIf` rule handles identity for all git operations **inside an existing repo**. But during `git clone`, there's no `.git` directory yet, so it doesn't apply. We fix this with a shell function already included in the `.zshrc` above:

```zsh
git() {
  if [[ "$1" == "clone" && "$PWD" == "$HOME/office"* ]]; then
    shift
    local rewritten=("${@/git@github.com:/git@github-office:}")
    command git clone "${rewritten[@]}"
  else
    command git "$@"
  fi
}
```

What this does:

- Intercepts any `git clone` call made from inside `~/office/`
- Rewrites `git@github.com:` to `git@github-office:` in the URL
- Passes the rewritten URL to the real git binary
- Every other git command passes through completely untouched

#### 9. Verify Setup

```bash
# Test cloning a personal repo (from anywhere outside ~/office)
cd ~
git clone git@github.com:your-personal-username/some-repo.git

# Test cloning a work repo (from inside ~/office)
cd ~/office
git clone git@github.com:your-org/private-repo.git

# Check identity inside a personal repo
cd ~/some-repo
git config user.email   # should show personal email

# Check identity inside an office repo
cd ~/office/private-repo
git config user.email   # should show work email
```

### How It All Works Together

| Stage | What handles it | Result |
|---|---|---|
| `git clone` inside `~/office/` | Shell function rewrites URL | SSH uses `~/.ssh/id_office` |
| Any git op on a repo in `~/office/` | `includeIf` loads `~/.gitconfig-office` | Work email on commits, office SSH key |
| Everything outside `~/office/` | Global `~/.gitconfig` only | Personal email, personal SSH key |
| New terminal session | `ssh-agent` block in `.zshrc` | Both keys are loaded automatically |

## Working with WSL in VS Code

After installing WSL and the WSL extension in VS Code, the next step is using it the right way.  

A common mistake is opening projects from the Windows filesystem and assuming they are running in WSL.  
That's not true.  

To properly use WSL, your project must live **inside the Linux filesystem**, not inside `C:`.

### Open a Project in WSL

1. Open VS Code
2. Press `Ctrl + Shift + P`
3. Search for `WSL: Open Folder in WSL`
4. Select a folder inside Linux

Once opened, you'll see **WSL: Ubuntu** in the bottom-left corner.

### Alternative Way

You can also open a project directly from the WSL terminal:

```bash
cd ~/.../your-project
code .
```

### Important Note

Avoid working inside:

```
/mnt/c/Users/...
```

Always work inside:

```
/home/your-username/
```

Working from Windows folders slows things down and can cause permission issues.  
Also, always commit your code before experimenting with WSL.  
If something breaks, recovery can be difficult.
